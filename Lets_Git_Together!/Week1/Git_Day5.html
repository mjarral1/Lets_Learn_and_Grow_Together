Working Across Teams

On Day 5, Mary, Jane, and Raja were tasked with collaborating with another team on a shared GitHub repository. To their surprise, the repository didn’t allow them to push changes directly. Instead, they had to use forks, clones, and pull requests to contribute.
Join the trio as they navigate this new collaboration model and master advanced GitHub workflows.


Task 1: Forking the Repository
One of the senior software developers explained the tasks to the team:
"The repository you’ll be working on belongs to another team, so you won’t have direct push access.
You’ll need to fork it to your own GitHub accounts to start contributing."
"Fork? What’s that?" asked Raja.
Mary answered, "A fork is like creating your personal copy of the repository on GitHub.
You can make changes there without affecting the original project."
Following their senior’s instructions, each team member forked the repository by clicking the Fork button on the project’s GitHub page.

Task 2: Cloning the Fork
Next, they cloned their forks to their local machines:
git clone
Raja typed: 

git clone https://github.com/raja123/forked-repo.git

Jane added, "Now we can work on this project locally, just like we did with our own repositories."

Task 3: Making Changes
Each team member switched to a new branch to work on their tasks:
git checkout -b feature-update
Mary fixed a typo in the project documentation.
Raja improved a function in the code.
Jane added unit tests for the new function.
They all staged and committed their changes:
git add .
git commit -m "Fix typo in documentation"

Task 4: Pushing Changes to the Fork
"Since we’re working on forks, we’ll push our changes to our own repositories," said Mary.
Each team member pushed their changes:
git push origin feature-update

Task 5: Creating a Pull Request
"Now comes the exciting part: contributing to the original repository using a pull request!" exclaimed Jane.
They went to their GitHub forks, clicked on the Pull Requests tab, and selected New Pull Request.
They compared their branches to the original repository’s main branch and submitted pull requests with clear descriptions of their changes.

Task 6: Reviewing and Merging Pull Requests
Mary asked, "What if the original repository gets updated after we fork it? How do we stay up to date?"
Excellent question! Then senior developers introduced them to the concept of the upstream repository:
git remote add upstream
git fetch upstream
git merge upstream/main
Mary practiced syncing her fork and shared her updates with the team.

Senior developer then concluded today's lesson for the team:
1. Forks let you work on repositories you don’t own.
2.Clones bring remote repositories to your local machine.
3.Pull requests enable you to contribute changes back to the original repository.
4.Upstream syncing ensures your fork stays up to date with the original repository.

Would you like additional examples or clarification on any part? Senior Developer asked to the team.
Mary, Jane, and Raja thumbs up and started to review the tasks of the day!
