Task 1: Checking the Status

Mary started her day by making some changes to the project‚Äôs HTML file to improve the header design. She wanted to ensure her changes were properly tracked before committing them. She ran the following command: 

 git status

"Hey team, I see the header file I edited is showing up as modified. Let me stage it!" said Mary. Mary staged her changes: 
git add header.html

Then, she committed them: 
git commit -m "Updated header design"

Mary recalled in her mind that git statushelps check which files are modified, staged, or untracked, making it easier to manage changes.

Task 2: Checking the Project History

Jane wanted to see all the commits made so far in the project. She used the following command:

"Wow, this shows a clean summary of our commits!" Jane exclaimed. She noticed that Mary‚Äôs "Updated header design" commit was the latest. Jane then ran:

to see the detailed commit history with author names and timestamps. She murmured that the

command provides insights into the project's progress and who made changes. and to show changes between working directory and index or between commits, use this command:

Task 3: Trouble with Merging

Raja was working on a new feature in a branch called feature-footer . Meanwhile, Mary updated the same part of the HTML file in the main branch. When Raja tried to merge his branch into main, he ran into an issue:
git checkout main git merge feature-footer
Git threw a warning:
CONFLICT (content): Merge conflict in footer.html
"Uh-oh! A merge conflict," Raja said nervously. "But no worries, I can handle this."
What Raja Did:
1. Opened the footer.html file.
2. Found the conflict markers 

<<<<<<< HEAD
        
Updated by Mary

         =======
         
New feature by Raja

         >>>>>>> feature-footer

3. Edited the file to merge the changes:
Updated by Mary and enhanced by Raja

4. Staged the resolved file:
git add footer.html
5. Committed the changes: git commit -m "Resolved merge conflict in footer.html"
"Great work, üôå Raja!" Mary and Jane cheered. so, they understood that Merge conflicts occur when Git doesn‚Äôt know which change to keep. Resolving them involves carefully reviewing and editing the conflicting files.

Task :4 Pushing Changes to GitHub

Once the conflict was resolved, Raja pushed the updated branch to GitHub by using the command:

The team reviewed the changes on GitHub and confirmed everything was working fine. So, üìù remember always push your changes after resolving conflicts to keep the remote repository up-to-date.
